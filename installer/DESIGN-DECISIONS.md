# installer/DESIGN-DECISIONS.md
# ENS-81 resolved decisions — recorded 2026-04-24

## §1. CLI-only install — service binaries excluded from install.sh

**Decision:** `install.sh` fetches ONLY the `enscrive` CLI binary.

The three service binaries (`enscrive-developer`, `enscrive-observe`,
`enscrive-embed`) are NOT installed by `install.sh`.  They are fetched
separately by `enscrive init --mode self-managed`, which consumes the
cross-repo manifest published by the Factory Squad (ENS-91).

**Rationale:**
- A user running managed mode (`enscrive init --mode managed`) against
  `api.enscrive.io` does not need the service binaries at all.
- Downloading ~100 MB of service binaries on every fresh CLI install is
  wasteful for the majority of users.
- `install.sh` consumes only the `binaries.enscrive` entry from the ENS-91
  manifest; `enscrive init --mode self-managed` consumes the full manifest.

**References:** ENS-81 ticket scope, `enscrive-governance/plans/RELEASE-INDUSTRIALIZATION-2026-04-23/DESIGN.md` §2.6.

## §2. Dev channel URL for the install one-liner

**Decision:** The install one-liner uses `https://dev.enscrive.io/install`
during beta.

Production CloudFront at `enscrive.io/install` is not yet provisioned
(pending CLI-REL-010 / ENS-81 GA sub-ticket).  The dev distribution
(`EWE9BH1POOS0A`, bucket `enscrive-install-artifacts-dev`) is live and
operational.

**Production follow-up:** once `enscrive.io/install` is provisioned, update:
1. `installer/install.sh` default manifest URL from dev to prod pointer.
2. `README.md` install snippet.
3. S3 publish target in the team-lead TODO comment.

## §3. Manifest source of truth: ENS-91 cross-repo manifest, not installer/manifests/

**Decision:** The stale `installer/manifests/prod/latest.json` placeholder
was deleted.  `installer/scripts/generate_manifest.py` was also deleted.

The ENS-91 cross-repo manifest (generated by the Factory Squad's
`manifest.yml` workflow and published to S3/CloudFront) is the authoritative
source.  There is no separate in-repo prod manifest to maintain.

`installer/manifests/dev/` contains a schema-reference fixture for offline
testing only — it is not served as a live manifest.

---

## §4. cosign verification — optional scaffold (ENS-82)

**Decision:** `install.sh` checks for a `.bundle` file next to the binary URL
and verifies it with `cosign verify-blob` if cosign is on PATH.  If the bundle
is absent or cosign is missing, the installer warns but does not fail.

This is expected behavior during beta: ENS-82 (cosign signing workflow) has
not yet been dispatched against a v0.1.0-beta.1 tag.  Once ENS-82 cuts its
first signed release, `.bundle` files will be present and cosign verification
will run automatically for users who have cosign installed.

`--insecure` unconditionally skips cosign verification.
